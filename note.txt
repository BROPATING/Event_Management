    /auth     → Floor 1 → handles register, login

auth.controller.ts
    this is where register and login logic lives.
    Request → Route → Controller → Database → Response

auth.middleware.ts
    Think of it as a bouncer at a club. Every request trying to access a protected route must show 
    their token (ID card). 

error.middleware.ts
    No matter where an error happens in your app — auth, events, bookings, reports — it always falls down to this one 
    place which sends a clean response back to the client.
    | Code | Meaning | Used when |
    |---|---|---|
    | `200` | OK | Successful GET request |
    | `201` | Created | New resource created |
    | `400` | Bad Request | Invalid input or business rule violation |
    | `401` | Unauthorized | No token or invalid token |
    | `403` | Forbidden | Valid token but wrong permissions |
    | `404` | Not Found | Resource doesn't exist |
    | `409` | Conflict | Email already in use |
    | `500` | Internal Server Error | Unexpected crash |

Summary — Your middleware chain

Every request passes through:

1. express.json()    → parse body
2. authMiddleware    → verify token (protected routes only)
3. controller        → business logic
4. errorMiddleware   → catch any errors (last resort)
    
event.controller.ts
    We perform CRUD operation 
    C → Create  → POST   /events
    R → Read    → GET    /events and /events/:id
    U → Update  → PUT    /events/:id
    D → Delete  → DELETE /events/:id (cancel)

booking.controller.ts
    this is the most important controller because it uses transactions.
    Bookings involve two database operations that must happen together:
    1. Deduct seats from tier    (update tier)
    2. Create booking record     (insert booking)
    If operation 1 succeeds but operation 2 fails — seats are deducted but no booking exists. 
    That's a data corruption bug. Transactions solve this.
