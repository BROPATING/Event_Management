🎭 The Scenario

You have been hired as a backend developer for Evently — a growing startup that lets organizers create and manage public events, and lets attendees browse and book tickets. The frontend team is not ready yet, so your job is to build the entire REST API that they will eventually consume.

The business wants the system to support multiple ticket tiers per event (e.g., General Admission, VIP, Early Bird), track how many tickets are sold vs available, and allow the finance team to run revenue and attendance reports at any point.

📌 Important There is no predefined database schema. Reading the requirements and the analytical questions below is how you discover what your schema must look like. Design first, build second.

✅ Functional Requirements

1. Authentication

  • A user can register with a name, email, and password. Passwords must be hashed (use bcrypt).
  • A user can log in and receive a signed JWT access token.
  • All routes except POST /auth/register and POST /auth/login require a valid JWT in the Authorization: Bearer <token> header.

2. Organizers & Events

  • Any authenticated user can become an organizer by creating an event.
  • An event must have: title, description, venue name, city, start date/time, end date/time, and a status (draft, published, cancelled).
  • Only the organizer who created an event can update or cancel it.
  • An event can have multiple ticket tiers. Each tier has: a name, a price, and a total capacity (number of seats available).
  • An organizer can add, edit, or remove ticket tiers — but not after any ticket in that tier has been sold.
  • Cancelling an event automatically sets all its ticket tiers as unavailable.

3. Ticket Bookings

  • An authenticated user can book one or more tickets of a given tier in a single request.
  • Booking must be wrapped in a TypeORM transaction: deduct available seats and create the booking record atomically.
  • If the requested quantity exceeds available seats, the booking must fail gracefully with an appropriate error message — no partial bookings.
  • A user can view all their own bookings.
  • A user can cancel their booking — seats must be returned to the tier's available count (also in a transaction).
  • Bookings for a cancelled event cannot be made.

4. Reporting Endpoints

The following reporting endpoints must use the TypeORM Query Builder — no raw SQL strings, no repository .find(). These endpoints exist specifically to answer the analytical questions in the next section.

  • Revenue summary per event (total revenue, total tickets sold) — filterable by date range.
  • Attendance summary per city — filterable by month and year.
  • Top-selling events overall — with a configurable limit query param (default 5).
  • Breakdown of bookings per ticket tier for a given event.
  • List of events that still have available seats, sorted by soonest start date.

🔍 Analytical Questions

Your reporting endpoints must be capable of answering all of the following questions. Read them before you design your schema — they will reveal what data you need to store and how entities must relate to each other.

🚨 These questions are part of your submission. Each Postman request in the Reports folder must be named after the question it answers, and must return a response that clearly addresses it.
Question 1

What is the total revenue generated from all events held in January 2025?

Hint: revenue = ticket price × quantity booked. Think about what "held in January" means for your date filtering.
Question 2

Which city had the highest number of tickets sold across all events in Q1 2025 (January – March)?

Question 3

What are the top 3 best-selling events of all time, ranked by total tickets sold?

Question 4

For event "Tech Summit 2025", how many tickets were sold per tier, and what is the revenue contribution of each tier as a percentage of the event's total revenue?

Hint: this requires aggregation + arithmetic in your query builder output or computed in the controller.
Question 5

Which events currently have fewer than 10 seats remaining across all their tiers combined?

Question 6

What is the average ticket price across all tiers of all published events, grouped by city?

Question 7

How many unique attendees (distinct users) bought tickets to events organised by a specific organiser?

Hint: "unique" means count distinct user IDs across all bookings for that organiser's events.
Question 8

What is the total number of cancelled bookings per event, and how much potential revenue was lost as a result?

Question 9

Which ticket tier type (by name, e.g., "VIP", "General Admission") generates the most revenue across all events combined?

Hint: tier names may repeat across events — group by tier name, not tier ID.
Question 10

List all events that are published but have zero bookings so far — sorted by start date ascending.

Question 11

What is the month-over-month revenue trend for the entire platform in 2025? Return one row per month showing total revenue.

Question 12

Which user has spent the most money on tickets in total (across all their non-cancelled bookings)?

Question 13

For events happening in the next 30 days, what is the percentage of seats already sold per event?

Hint: this involves comparing sold seats to total capacity across all tiers of each event.

📮 Postman Collection — Submission Guide

📄 Read the companion document first Before building your collection, read postman-tutorial.html (sent alongside this document). It covers everything you need: folders, variables, scripts, and how to export — step by step with screenshots. Come back here once you've read it.

What your collection must contain

  • Folders for each group: Auth, Events, Tiers, Bookings, Reports.
  • Every endpoint you build must have at least one request — with a realistic example body where applicable.
  • The Reports folder must have one request per analytical question (Q1–Q13), named clearly — e.g. "Q4 — Revenue % per tier for Tech Summit 2025".
  • A Collection Variable named baseUrl set to http://localhost:3000.
  • A Collection Variable named token — populated automatically by the Login request script (see tutorial).
  • All protected requests must use Authorization: Bearer {{token}} in their headers.

Exporting your collection

  1. In Postman, click the three-dot menu (⋯) next to your collection name.
  2. Select Export → Collection v2.1.
  3. Save the file as evently-collection.json — you will include this in your zip.
✅ Before submitting Use the Postman Collection Runner (▶ Run button on your collection) to run all requests top-to-bottom. Fix anything that fails. A clean run end-to-end is the goal.

Submit a single .zip file named evently-yourname.zip containing:

Your project folder All source code — delete node_modules/ before zipping. The reviewer will run npm install themselves.

🚀 Getting Started Checklist

  • Read all requirements and analytical questions in full before writing any code
  • Sketch your entity diagram on paper — identify all relations
  • Initialise the project: npm init -y, install dependencies
  • Configure TypeORM with SQLite and migrations folder
  • Create all entities and run your first migration
  • Build auth routes + JWT middleware
  • Build events and tiers CRUD
  • Build bookings with transactions
  • Build reporting endpoints (Query Builder)
  • Seed realistic test data for at least 3 events, 2 organisers, 5 users
  • Build and run Postman collection — verify all 13 questions return data
  • Export Postman collection as evently-collection.json
  • Delete node_modules/ from your project folder
  • Zip the project folder + evently-collection.json together into evently-yourname.zip
  • Submit the zip file
⚠️ Reminder on synchronize: true You may use synchronize: true during early development to iterate quickly on your schema, but your final submission must use proper migrations and synchronize: false. Reviewers will run npm run migration:run to set up the database from scratch.
