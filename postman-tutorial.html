📮 Postman Tutorial

Folders · Variables · Scripts · Export — everything you need for your submission

📌 Who this is for This tutorial assumes you already know how to open Postman, create a Collection, and send a basic GET or POST request. Everything beyond that is covered here from scratch.

By the end of this document you will be able to:

  • Organise requests into folders inside a collection
  • Understand what variables are and create them at the collection level
  • Use variables inside your request URLs and headers with the {{variableName}} syntax
  • Write a simple Post-response script that saves your JWT token automatically after login
  • Export your collection as a .json file ready for submission

1 Folders — Organising Requests

What is a folder?

A folder (Postman calls them folders or groups) is just a named container that sits inside a collection. Think of it like a subfolder on your desktop. Instead of having 20 requests all jumbled together, you group them by feature: all auth requests in one folder, all event requests in another, and so on.

Example structure for this project
📁 Evently API  ← your Collection
  📂 Auth
      POST  /auth/register
      POST  /auth/login
  📂 Events
      GET   /events
      POST  /events
      PUT   /events/:id
      ...
  📂 Tiers
      POST  /events/:id/tiers
      ...
  📂 Bookings
      POST  /bookings
      ...
  📂 Reports
      GET   /reports/revenue       ← Q1
      GET   /reports/cities        ← Q2
      ...

How to create a folder

  1. In the left sidebar, find your collection. Hover over its name — three dots (⋯) appear to the right of it.

  2. Click the (⋯) menu and choose Add folder.

  3. A new folder appears inside your collection. Type its name — e.g. Auth — and press Enter.

  4. Repeat for every group: Events, Tiers, Bookings, Reports.

How to add a request into a folder

  1. Hover over the folder name (not the collection). The (⋯) appears again.

  2. Click (⋯) → Add request. A new blank request appears inside that folder.

  3. Name the request clearly — e.g. Register User or Q1 — Total Revenue Jan 2025.

⚠️ Moving existing requests If you already created requests outside any folder, you can drag and drop them into a folder directly in the sidebar. Just click and hold the request, then drag it on top of the folder name until the folder highlights, then release.

2 Variables — Stop Copy-Pasting

What is a variable?

A variable is a named placeholder that stores a value you want to reuse across many requests. Instead of typing http://localhost:3000 in every single request URL, you store it once in a variable called baseUrl, and then every request just references {{baseUrl}}.

If you ever change your server port, you update it in one place and every request updates automatically.

The two variables you need for this project
Variable nameValuePurpose
baseUrlhttp://localhost:3000Your server's base address — used in every URL
token(filled automatically by script)The JWT returned after login — used in auth headers

Variable scopes — where does a variable live?

Postman has several places where variables can be stored. The most important ones are:

ScopeWhere it's setWhen to use it
Global Postman app settings Rarely — shared across all collections
Collection Your collection's Variables tab ✅ Use this — scoped to your project only
Environment A named Environment (e.g. "Dev", "Prod") Useful for switching between servers; not needed here

For this project, use Collection variables — they travel with the collection when you export it, which is exactly what you want for submission.

How to create collection variables

  1. Click on your collection name in the sidebar (not a folder, the root collection). The collection overview opens in the main panel.

  2. Click the Variables tab at the top of the panel (you'll see tabs like Overview, Authorization, Pre-request Script, Tests, Variables).

  3. You'll see a table with two columns: Variable and Initial Value (and Current Value).

  4. Click the empty row under Variable and type baseUrl. Then click under Initial Value and type http://localhost:3000. Press Tab — the Current Value fills in automatically.

  5. Click the next empty row. Add a second variable: name it token, leave its value blank for now — it will be filled in by a script after you log in.

  6. Click Save (Ctrl+S / Cmd+S).

📌 Initial Value vs Current Value Initial Value is what gets exported with the collection — anyone who imports your collection sees this. Current Value is what Postman actually uses when sending requests on your machine. They start the same, but scripts (Section 3) update only the Current Value at runtime. Never put a real secret in Initial Value — it will be in the exported file.

How to use a variable in a request

Anywhere in Postman — a URL, a header value, a body field — you reference a variable by wrapping its name in double curly braces: {{variableName}}.

Example — using baseUrl in a URL:

Instead of this…
http://localhost:3000/auth/login
Write this
{{baseUrl}}/auth/login

Postman highlights variable names in orange when it recognises them. If the variable name stays dark/red, check your spelling.

Example — using token in the Authorization header:

  1. Open any protected request (e.g. GET /events).

  2. Click the Headers tab.

  3. Add a new header row:

    Key: Authorization    Value: Bearer {{token}}

That's it. Once the token variable has a value (set by the login script in Section 3), every request using Bearer {{token}} will automatically send the correct JWT.

3 Scripts — Saving the JWT Automatically

What is a Post-response script?

A Post-response script (also called a Test script in older Postman versions) is a small piece of JavaScript that Postman runs after it receives a response. You can use it to read data from the response and store it somewhere — like saving the JWT token into a collection variable so all your other requests can use it automatically.

Without this script, you would have to:

  1. Send the login request.
  2. Copy the token from the response body.
  3. Manually paste it into every other request's Authorization header.
  4. Repeat every time the token expires.

With the script, step 1 is all you need. Postman handles the rest.

How to add the script to your Login request

  1. Open your POST /auth/login request.

  2. Click the Scripts tab at the top of the request panel.

    You will see two sub-tabs: Pre-request and Post-response. Click Post-response.

  3. You'll see an empty code editor. Paste the following code exactly as written:

const res = pm.response.json();

if (res.token) {
    pm.collectionVariables.set("token", res.token);
    console.log("✅ Token saved:", res.token);
} else {
    console.warn("⚠️ No token found in response. Check your login endpoint.");
}

  1. Press Ctrl+S / Cmd+S to save.

  2. Send a login request with valid credentials. In the response, click Console at the bottom of the Postman window — you should see ✅ Token saved: eyJ...

  3. Go back to your collection's Variables tab. The Current Value of token should now show the JWT string. Leave Initial Value blank.

✅ You're done Every request in your collection that uses Bearer {{token}} in its Authorization header will now automatically use the fresh token from your last login — no manual work needed.

Understanding the script line by line

LineWhat it does
pm.response.json() Parses the response body as JSON and returns it as a JavaScript object
res.token Reads the token field from that object — adjust this if your API returns res.accessToken or similar
pm.collectionVariables.set("token", res.token) Stores the value into the collection variable named token
console.log(...) Prints to the Postman Console (bottom bar) — useful for debugging
⚠️ If your API returns a different field name If your login endpoint returns { "accessToken": "eyJ..." } instead of { "token": "eyJ..." }, change res.token to res.accessToken in the script. Match whatever your API actually returns.

4 Full Request Checklist

Before moving on, verify every request in your collection follows these rules:

  • URL uses {{baseUrl}} — e.g. {{baseUrl}}/events
  • All protected requests have Authorization: Bearer {{token}} in their Headers tab
  • POST / PUT requests have a JSON body with realistic example data (not empty)
  • Report requests (Q1–Q13) are named clearly with the question number
  • The Login request has the Post-response script added
  • Sending Login first, then any protected request, works without errors

5 Exporting Your Collection

Exporting produces a single .json file that contains your entire collection — all folders, requests, variables, and scripts. This is what you submit.

  1. In the sidebar, hover over your collection name. Click the (⋯) three-dot menu.

  2. Choose Export from the dropdown.

  3. A dialog appears asking which format to use. Select Collection v2.1 (this is the modern standard — always use this).

  4. Click Export. A file save dialog opens.

  5. Save the file as evently-collection.json. Remember where you saved it.

  6. Submit this file together with your GitHub repository link.

📌 What gets exported — and what doesn't The export includes: all requests, folders, headers, body examples, scripts, and variable Initial Values. It does not include Current Values of variables (those are local to your machine). This is why the token variable's Initial Value should remain blank — its value is generated at runtime by the script.

6 Running the Whole Collection (Sanity Check)

The Collection Runner lets you run every request in your collection in sequence, automatically, with one click. Use this to check that everything works before submitting.

  1. Click on your collection name in the sidebar to open the collection overview.

  2. Click the ▶ Run button near the top right of the panel.

  3. The Collection Runner opens. Make sure your requests are ordered logically: Register → Login → everything else. You can drag requests to reorder them here.

  4. Click Run Evently API (or whatever your collection is named).

  5. Postman runs each request one by one. You will see a green ✅ or red ❌ next to each one.

  6. Fix any red requests before exporting and submitting.

⚠️ Order matters in the Runner The Login request must run before any protected request, because the script on Login is what populates the token variable. If you run a protected request first, the token will be empty and you'll get a 401.
